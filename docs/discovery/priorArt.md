## SUMMARY

Surveyed 19 Claude Code observability/monitoring/orchestration OSS projects plus Anthropic's own built-in "agent view." Nothing combines all four target features — live spatial multi-agent graph, per-agent animated character/avatar states, embedded live terminal output, and replay — in one product. The two closest analogs are patoles/agent-flow (1.3k★, live node-graph canvas + JSONL replay, but abstract flow-diagram nodes, no character/avatar layer) and hoangsonww/Claude-Code-Agent-Monitor (813★, has DAG/Sankey views, session replay, and a single decorative mascot cat — but not per-agent animated status characters). That gap is real and defensible. For naming, of the 15 candidates checked against GitHub + npm registry, 9 are hard collisions (crewdeck and agentdeck are the worst — crewdeck is literally an npm package + commercial product doing "visual dashboard for Claude Code agents"; antfarm, agentarium, waggle, overseer, termhive, opsdeck, formicary, colony all have real, actively-maintained same-space competitors, several with hundreds to thousands of stars). The clean top 3 are murmuration, swarmscope, and swarmdeck.

## FINDINGS

## 1. PRIOR ART

### Tier A — Direct competitors (closest to our concept)

**patoles/agent-flow** — 1,275★, actively pushed (Jul 11 2026). Integration: hybrid — lightweight HTTP hooks pushed directly from Claude Code sessions ("zero-latency streaming") + Codex `~/.codex/sessions/**/rollout-*.jsonl` tailing + accepts arbitrary JSONL event logs for replay. Visuals: interactive node-graph canvas (pan/zoom/click-to-inspect), tool-call branching/return-flow animation, timeline+transcript side panels, file-attention heatmap, multi-session tabs. **Has replay** (point at a JSONL log and "replay or watch"). **What it lacks**: no per-agent animated character/avatar — nodes are abstract tool-call boxes, not agent "workers" with visible states (idle/thinking/blocked/done); no stated mission-control narrative framing; terminal output lives in a side panel, not inline in the graph node.

**hoangsonww/Claude-Code-Agent-Monitor** — 813★, pushed today (Jul 17 2026). Integration: native Claude Code hooks (tool-use/session events) + JSONL transcript parsing for historical import + subagent `subagents/agent-*.jsonl` file scanning + WebSocket push. Electron native apps for macOS/Windows plus web UI. Visuals: dark dashboard, Kanban board, subagent hierarchy tree, DAG + Sankey diagrams, session replay via transcript viewer, token/cost analytics, **and a floating mascot cat ("Tabby")** for UI flourish — the only prior-art project with any character element at all. **What it lacks**: Tabby is a single fixed decorative companion, not one animated character per agent whose pose/expression reflects that specific agent's live hook-driven status; the DAG/Sankey views are static analytical diagrams, not a persistent live spatial canvas.

**disler/claude-code-hooks-multi-agent-observability** — 1,493★, well-known via the IndyDevDan/disler YouTube channel. Integration: pure hooks — 12 Claude Code hook events (PreToolUse, PostToolUse, SessionStart, SubagentStart, etc.) captured by Python/uv scripts → HTTP POST → Bun server → SQLite → WebSocket → Vue 3 client. Visuals: real-time timeline with dual session/app color coding, a canvas-based "live pulse chart" (activity density + tool emojis), chat transcript viewer, dark/light theme, "smooth animations." **What it lacks**: no spatial graph layout at all (timeline/pulse-chart, not nodes-and-edges), no character states; explicit replay isn't implemented (only paginated `GET /events/recent` history).

**simple10/agents-observe** — 625★, actively maintained (28 releases, Docker-packaged, installable as a Claude Code plugin-marketplace plugin). Integration: pure hooks — `.claude/settings.json` hooks fire `observe_cli.mjs`, which POSTs an envelope to a Hono/Node server (SQLite) → WebSocket → React 19 dashboard. Visuals: swim lanes per agent, parent/child subagent hierarchy, timeline navigation, **full session replay** with human-readable session names (e.g. "twinkly-hugging-dragon"), token/cost breakdowns. **What it lacks**: swim-lane/list layout, not a spatial node graph; no animated character states.

**Anthropic's own built-in "agent view"** (code.claude.com/docs/en/agent-view) — Claude Code ships a native multi-agent management view with color-coded status (green done / yellow needs input / blue active). Not third-party prior art, but the single biggest platform risk: Anthropic controls this roadmap and could extend it to compete directly with any third-party visualization layer.

### Tier B — Session/process managers (no visualization graph; PTY or process wrappers)

- **opcode** (formerly claudia, winfunc/opcode) — 22,178★, the most popular Claude Code GUI by far, but stale (no release since Aug 2025, unanswered issues for ~7 months). Integration: Tauri 2 (Rust) desktop app that wraps/spawns the `claude` CLI process directly (PTY-style process management), with checkpoints, custom agents, background agents. Lacks any live multi-agent graph — it's a single-session chat-style GUI.
- **Crystal** (stravu/crystal, now deprecated → rebranded "Nimbalyst" as of Feb 2026) — 3,098★. Electron desktop app spawning Claude Code/Codex processes into isolated git worktrees, SQLite session persistence, session archiving/templates. Text/list based, no graph, no animation.
- **Vibe Kanban** (BloopAI) — 27,421★, the single most-starred project surveyed, but is currently sunsetting (announced wind-down; codebase remains available). Integration: Rust backend spawns agent CLI processes (Claude Code, Codex, Gemini, Copilot, Cursor, Amp, OpenCode, and more) per kanban card/workspace, each with its own git branch, terminal, and dev server. Kanban-card metaphor, not a spatial graph; no character animation.
- **omnara** (YC S25) — 2,651★. Runs Claude Code headless (process wrapper) and relays to a mobile/web client via Omnara's own backend; GitHub Actions repository-dispatch integration for remote triggers. Remote-control/chat framing, not a multi-agent visual canvas.
- **happy / happy-coder** (slopus/happy 22,703★, slopus/happy-cli 557★) — happy-cli wraps a Claude Code session (process wrapper) and streams it E2E-encrypted to iOS/Android/web clients with QR pairing and realtime voice. Remote-control mobile client, single-session focus, not a graph.
- **CCManager** (kbwo/ccmanager) — 1,194★. Ink-based TUI spawning 8 different agent CLI types (Claude Code, Gemini, Codex, Cursor Agent, Copilot, Cline, OpenCode, Kimi) into git worktrees with waiting/busy/idle status; auto-approval, devcontainer support. Terminal-only, no browser canvas.
- **claude-squad** (smtg-ai/claude-squad) — 8,130★. Go/tmux-based TUI spawning multiple agent CLI processes (Claude Code, Codex, Gemini, Aider, OpenCode, Amp) each into an isolated git worktree + tmux pane, single terminal window. Terminal-only, text status only.
- **Conductor** (conductor.build, Melty Labs) — closed-source Mac app (no public star count). Spawns Claude Code/Codex/Cursor agent processes into isolated git worktrees, list-style dashboard, GitHub/Linear API integration (open/comment/respond to PRs). List/dashboard view, not a spatial canvas; direct commercial (not OSS) comp worth tracking.

### Tier C — Log/transcript viewers and usage analytics (no live view at all)

- **sniffly** (chiphuyen/sniffly, by ML educator Chip Huyen) — 1,247★. Pure static ingestion of `~/.claude/projects` JSONL transcripts into a local Python/browser dashboard (usage stats, error analysis — e.g. its widely-shared stat that "Content Not Found" errors are 20-30% of all Claude Code errors). Post-hoc only, no live/real-time, no graph, no character.
- **claude-code-log** (daaain/claude-code-log) — 1,150★. Pure JSONL transcript parser → static HTML/Markdown export, plus an interactive TUI for browsing/resuming sessions. Archival, not live.
- **claude-trace** — fragmented name with no single canonical project: vexorkai/claude-trace and delexw/claude-code-trace are JSONL cost/log analyzers; badlogic/lemmy's `claude-trace` app is a request/response tracer; joemccann/claude-trace is actually an unrelated OS-level CPU/process monitor. None graph- or character-oriented.
- **ccusage** (ccusage/ccusage) — 17,243★, extremely popular `npx ccusage` one-liner reading local JSONL logs into daily/weekly/session cost tables (now supports Codex, OpenCode, Amp, Gemini CLI, etc. too, not just Claude Code). Purely tabular, zero visualization.
- **claude-code-webui** (sugyan/claude-code-webui) — 1,142★. Node/Deno server spawns the Claude CLI process (PTY-style) and streams chat responses to a responsive mobile-friendly web chat UI. Single-session chat interface, no multi-agent graph.
- **agent-sessions** (jazzyalex/agent-sessions) — 722★. Native Swift macOS app that reads local session-log files across many agent CLIs (Codex, Claude Code, OpenCode, Cursor, Copilot CLI, Pi, etc.) for browse/search/resume (shells out to the resume command). Retrospective transcript browser, not live.

### Conclusion on the core question

No surveyed project fuses a **live spatial multi-agent graph** + **per-agent animated character states** + **inline live terminal output per node** + **replay** into one coherent "mission control" narrative. agent-flow owns the closest technical shape (graph + replay + hooks) but stays abstract/diagrammatic. Claude-Code-Agent-Monitor owns the closest tonal shape (a mascot, plus DAG/Sankey, plus replay) but the character is decorative, not state-driven per agent. **Our differentiation**: combine agent-flow's live spatial canvas + replay with a game-like illustrated layer where each agent panel is an animated character whose pose/expression is driven directly by that agent's own Claude Code hook events (idle/thinking/coding/blocked/error/done), with that agent's live terminal output streaming inside its own node — framed explicitly as "mission control," a framing no competitor currently owns.

## TOP 3 RECOMMENDATION

1. **murmuration** — _"Watch your Claude Code agents flock together, live."_ A starling murmuration is the real-world phenomenon of many small individuals visibly moving as one coordinated, mesmerizing whole — the single best available visual metaphor for a live animated multi-agent canvas. Safest collision profile of all 15 (no real AI-agent-space competitor with any traction). Caveat: the bare npm package name is taken by an unrelated DB-migration tool, so ship the CLI/package under a scoped or lightly modified name (e.g. `@yourorg/murmuration` or `murmuration-cli`) while keeping "Murmuration" as the product/GitHub name.

2. **swarmscope** — _"A live window into your Claude Code agent swarm."_ Maximally literal and self-explanatory (swarm = many small workers, scope = an instrument for watching), easy to say and spell, npm name fully free. Only soft risk: a couple of zero-star hobby repos are independently building near-identical concepts under this exact name right now — worth a quick recheck before launch, but nothing currently blocks it.

3. **swarmdeck** — _"The mission-control deck for your Claude Code agent swarm."_ Combines the "swarm of workers" framing with "deck" (command console), npm free, and the only same-name GitHub hits are either irrelevant (Docker Swarm templates) or zero-traction. Caveat: the "-deck" suffix is heavily overused in this exact niche (agentdeck, crewdeck, opsdeck all collide on sibling names), which slightly dilutes distinctiveness even though this specific string is clean.

Backups worth keeping in reserve: **hivewatch** (great literal fit — a hive of watched workers — and clear in the coding-agent niche, but a real unrelated company already owns the name commercially) and **stagecrew** (the single cleanest collision profile of the whole list, slightly weaker thematic bullseye).

## 3. README / POSITIONING LESSONS FROM THE MOST POPULAR PRIOR ART

1. **Lead with one visceral, nameable pain, not a feature list.** ccusage's whole pitch is "you're spending money and have no idea where it went — one command, ten seconds, no signup." That framing (not "AI agent observability platform") is what drove 17k stars and normalized the `npx <tool>` one-liner pattern across the ecosystem.
2. **Zero-friction, local-first, no-telemetry is a repeated trust signal, not a footnote.** sniffly, agent-sessions, and happy all foreground "runs locally / no telemetry / stays on your machine" near the top of their README — call this out explicitly and early.
3. **Show, don't tell — a GIF above the fold wins launches.** agent-flow's Show HN title was literally _"a beautiful way to visualize Claude Code actions"_ — the pitch was the visual itself, not a feature bullet list.
4. **Frame visuals as a debugging/learning outcome, not eye-candy.** agent-flow explicitly sells "build intuition for better prompts" and "see where time is spent" — turning a pretty graph into a stated productivity outcome reads far better on HN/GitHub trending than "cool dashboard."
5. **A named architecture diagram builds contributor trust fast.** disler's `hooks → Bun → SQLite → WebSocket → Vue` pipeline diagram is widely forked/reused because contributors can see exactly where to extend it — include one in the README, not just in docs.
6. **Distribution via a credible existing audience matters as much as the product.** disler's traction is inseparable from the IndyDevDan YouTube channel; sniffly's from Chip Huyen's existing ML-education following. Organic GitHub trending alone rarely explains these star counts.
7. **Maintenance cadence is itself a feature, and its absence is now visibly costing opcode mindshare** — despite 22k stars, ~7 months with no release/issue response is actively cited in comparison articles as a reason to switch to Crystal/Nimbalyst. Commit to a visible release cadence from day one.
8. **Broaden beyond Claude Code once the concept proves out.** Nearly every winner (omnara, agent-flow, ccusage, agent-sessions, CCManager, claude-squad) now also supports Codex/Gemini/OpenCode/etc. — a natural, well-trodden "v2" expansion that widens the addressable audience without diluting the initial pitch.
9. **Even category leaders wind down.** Vibe Kanban (27k★, the single most-starred project surveyed) is currently sunsetting, and Crystal rebranded away entirely — scale alone doesn't guarantee survival; a clear, honestly-stated sustainability/maintenance story matters for anyone deciding whether to depend on a new tool.
10. **Meet developers on the surface they already live in.** Winners increasingly ship on multiple surfaces early — TUI _and_ web _and_ native desktop (Tauri/Electron) — rather than picking one; opcode/Crystal both shipped desktop apps, and Claude-Code-Agent-Monitor added native macOS/Windows builds on top of its web UI.

## RISKS

- **Platform risk from Anthropic itself**: Claude Code already ships a native, color-coded multi-agent "agent view" (code.claude.com/docs/en/agent-view). Anthropic controls this roadmap and could extend it to compete directly with any third-party visualization layer at any time — this is a stronger long-term risk than any OSS competitor found.
- **Naming space is actively contested in real time.** Several candidate names (swarmscope, swarmdeck) surfaced small, zero-star same-name GitHub repos in the exact target niche that were updated within hours of this research session — the "clear" verdicts above reflect a snapshot, not a guarantee; re-verify immediately before publishing.
- **npm registry name ≠ GitHub name availability**, and vice versa — several recommended names (murmuration, stagecrew) require either a scoped/modified npm package name or accepting an unrelated, low-traffic existing npm package sharing the bare word. This is a minor practical friction, not a legal blocker, but should be resolved in the setup/tooling phase.
- **"Character/animation" claim could not be fully verified for every fringe project.** Web search and README fetches cover the dominant, highest-traction projects thoroughly, but the long tail of small (0-2 star) repos surfaced during naming collision checks (e.g. various "agent-*" dashboards) were not individually audited for whether they secretly already have character/avatar animation — none of the README/description text for any of them mentions it, which is the strongest available signal, but it is inference from descriptions rather than direct UI inspection.
- **Star counts and "pushed_at" timestamps are live/volatile** (several changed or were freshly updated during the research session itself, e.g. TermHive, swarmscope forks) — treat all popularity figures in this report as a July 17, 2026 snapshot, not a fixed fact.
- **Trademark risk for hivewatch is real but outside-industry** — hivewatch.com is a functioning commercial company in security/infrastructure monitoring, a different vertical from dev tooling, so likely low legal risk for an OSS project, but this was not verified against an actual trademark registry (only via web search of the company's existence), so treat as a caution flag rather than a cleared verdict if the name is ever used for a commercial entity.
- **Conductor (conductor.build) is closed-source**, so its exact integration mechanism (hooks vs. pure PTY spawn) was inferred from marketing copy and blog posts, not verified against source code — treat that detail as lower-confidence than the OSS projects, which were checked directly against README/source descriptions.

## RECOMMENDATIONS

- Proceed with **murmuration** as the primary name candidate; **swarmscope** and **swarmdeck** are equally clean fallbacks if "murmuration" tests poorly for pronounceability/searchability with the target audience. Do a final live re-check of GitHub/npm immediately before publishing the first commit, since several adjacent names (swarmscope in particular) are being actively squatted by small hobby projects in real time.
- Register the chosen name's npm package, GitHub org/repo, and a matching domain in the same session to lock it in before any public mention — several of today's "clear" names (swarmdeck, swarmscope) had zero-star same-name repos appear or update within hours of this search, indicating this naming space is being actively mined right now.
- Explicitly position the product against **agent-flow** and **Claude-Code-Agent-Monitor** in the README/launch copy rather than ignoring them — cite the specific gap (no per-agent animated character states driven by live hook status; no inline live terminal per node) so early adopters and HN commenters don't have to ask "how is this different from X" themselves.
- Build the "mission control" narrative into the product's actual UI language (panel labels, status vocabulary) now, before a name like opsdeck's "Mission Control dashboard for OpenClaw" or a future Anthropic-shipped feature claims that exact framing more visibly.
- Adopt at least three of the README lessons directly: (1) a GIF/animated demo at the very top of the README, (2) an explicit "local-first, no telemetry" line near the top, (3) a named architecture diagram (hooks → transport → store → live UI) so contributors can extend it immediately.
- Plan for multi-agent-CLI support (Codex, Gemini CLI, OpenCode) as a stated v2 milestone in the README/roadmap even if v1 ships Claude-Code-only — this is the single most common expansion path among every successful comparator surveyed.
- Because Vibe Kanban (27k★) is sunsetting and Crystal rebranded away, do not assume any single "leader" is stable prior art to defer to — re-verify the competitive landscape again shortly before launch.
