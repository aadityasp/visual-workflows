## SUMMARY

Claude Code persists a live, incrementally-flushed JSONL transcript per session plus a constellation of side files (subagent transcripts, workflow run summaries, spilled tool output, and several cross-session index/registry files under `~/.claude/`). Every event a tailer needs — turns, tool calls/results, token usage, errors, interrupts, subagent spawns, and even live process status — is observable from disk, linked via a small set of consistent keys (`uuid`/`parentUuid`, `message.id`, `tool_use.id`/`tool_use_id`, `agentId`, `toolUseId`, `origin`). The one true gap is interactive approval/permission prompts: this entire dataset runs under `bypassPermissions`, so that flow was never exercised and its on-disk shape is unverified. Liveness was directly confirmed two ways: the active session's `.jsonl` grew in both size and line count between two `stat` calls, and `~/.claude/sessions/<pid>.json` showed `status:"busy"` for that exact session in real time.

## FINDINGS

# 1. Main session transcript (`<sessionId>.jsonl`)

Census across sampled files (`4febcea9-...jsonl` 351 lines, `0ced7ffb-...jsonl` 127 lines) via `jq -r .type | sort | uniq -c`. Observed `type` values, union across samples:
`ai-title`, `assistant`, `attachment`, `bridge-session`, `file-history-delta`, `file-history-snapshot`, `last-prompt`, `mode`, `permission-mode`, `queue-operation`, `system`, `user`.

Common envelope fields present on almost every line type (`assistant`/`user`/`system`/`attachment`): `parentUuid` (`<uuid>|null`), `uuid` (`<uuid>`), `isSidechain` (bool — **always `false`** in every top-level session file sampled; never found `true` inline anywhere), `sessionId`/`session_id` (`<uuid>`, duplicated under two key names on assistant lines), `timestamp` (`<ISO8601>`), `cwd` (`<path>`), `gitBranch` (`<string>`), `version` (`<semver>`), `entrypoint` (`"cli"`), `userType` (`"external"`).

## `type:"user"`

Keys: `cwd, entrypoint, gitBranch, isSidechain, message, origin, parentUuid, permissionMode, promptId, promptSource, sessionId, timestamp, type, userType, uuid, version` (+ optional `toolUseResult`, `interruptedMessageId`, `sourceToolAssistantUUID`).

- `message` = `{role:"user", content: <string> | <array>}`. Array item `.type` observed: `"text"`, `"image"`, `"tool_result"`.
  - `tool_result` item: `{type, tool_use_id:<toolu_id>, content: <string>|<array>, is_error?:bool}`. `content` array form holds items like `{type:"tool_reference", tool_name:<string>}` (used for `ToolSearch` results).
  - `image` item: `{type:"image", source:{type:"base64", media_type:<string>, data:<string>}}`.
- `origin` (object, optional) — tags who/what produced this user turn: `{kind:"human"}` | `{kind:"task-notification"}` | `{kind:"peer", from:<agentName>, senderTaskId:<agentId>}`. **This is the mechanism by which a background teammate's reply is delivered back into a session as an ordinary user turn.**
- `promptSource` values seen: `typed`, `queued`, `system`, `suggestion_accepted` (absent = ordinary).
- `promptId` (`<uuid>`) — groups all assistant output produced in response to one submitted prompt.
- `toolUseResult` (optional, object OR string) — a **structured**, tool-specific mirror of the tool_result, richer than `message.content[].tool_result.content`. Shapes observed by originating tool:
  - Bash: `{interrupted:bool, isImage:bool, noOutputExpected:bool, stdout:<string>, stderr:<string>, persistedOutputPath:<path>|null, persistedOutputSize:<number>|null}` — `interrupted:true` is the flag for a killed/ctrl-C'd command; `persistedOutputPath` points into the session's `tool-results/` spill dir.
  - Read: `{type, file:{filePath, content, numLines, startLine, totalLines}}`.
  - Edit/Write: `{type, filePath, originalFile, content|oldString/newString, replaceAll?, structuredPatch:[{oldStart,oldLines,newStart,newLines,lines}], userModified:bool}` — **this is the direct source for `agent_file_modified`.**
  - `ToolSearch`: `{matches:[<toolName>], query, total_deferred_tools}`.
  - `TaskCreate` (todo tool): `{task:{id, subject}}`.
  - `TaskUpdate` (todo tool): `{success:bool, taskId, updatedFields:[<string>], statusChange:{from, to}}` — **direct source for a todo-item status-changed event** (distinct from the `Agent`/subagent system — same tool-family name, different concept).
- `interruptedMessageId` (string, optional) — appears only on a synthetic follow-up `user` line whose `message.content[0].text` is the fixed system marker `"[Request interrupted by user]"`. Its value equals the `message.id` of the assistant turn that was interrupted (see streaming/interrupt mechanics below).

## `type:"assistant"`

Keys: `cwd, entrypoint, gitBranch, isSidechain, message, parentUuid, requestId, sessionId, session_id, timestamp, type, userType, uuid, version`.
`message` keys: `content, diagnostics, id, model, role, stop_details, stop_reason, stop_sequence, type, usage`.

- `model` = e.g. `<model-id-string>` (real values seen include internal names like `claude-fable-5`, `claude-sonnet-5`, `claude-opus-4-8`, `claude-opus-4-8[1m]`) — **this is where model name lives.**
- `content[]` block `.type` census: `text`, `thinking`, `tool_use` (no `redacted_thinking` seen in sample).
  - `thinking`: `{type, thinking:<string>, signature:<opaque-string>}`.
  - `text`: `{type, text:<string>}`.
  - `tool_use`: `{type, id:<toolu_id>, name:<toolName>, input:<object>, caller:{type:"direct"}}` (`caller` only ever seen as `{"type":"direct"}` in this data; presumably distinguishes future indirect/inferred calls).
- `stop_reason` values seen: `"end_turn"`, `"tool_use"`, `"stop_sequence"`, `null`. **`null` occurs on every content-block line that is not the final block of a logical API turn** (see below) — a `null` stop_reason alone is not itself an interrupt signal.
- `usage`: `{input_tokens, output_tokens, cache_creation_input_tokens, cache_read_input_tokens, server_tool_use:{web_search_requests, web_fetch_requests}, service_tier, cache_creation:{ephemeral_1h_input_tokens, ephemeral_5m_input_tokens}, inference_geo, iterations:[{...same shape...}], speed}` — **this is where token usage lives**, duplicated identically across every content-block line of the same logical turn (see below), so a tailer must dedupe by `message.id`, not sum blindly.
- `diagnostics`: usually `null`; when present: `{cache_miss_reason:{type:"messages_changed", cache_missed_input_tokens:<number>}}`.

### Critical structural fact: one API "message" = multiple JSONL lines

A single logical assistant turn (one `message.id`, e.g. `msg_...`) is split across **multiple consecutive JSONL lines, one per content block**, each with its own `uuid`, chained via `parentUuid` to the previous block's `uuid`. Verified directly: a `message.id` with 4 content blocks (`thinking`→`text`→`tool_use`→`tool_use`) produced 4 separate JSONL lines with matching `message.id`/`usage`/`stop_reason` but distinct `uuid`/`parentUuid`/single-item `content`. **A tailer reconstructing one assistant turn must group consecutive assistant lines by `message.id` and concatenate `content` in file order**, not treat every JSONL line as an independent turn.

### Interrupt mechanics (directly observed)

When a user interrupts mid-stream: every content-block line belonging to that `message.id` is left with `stop_reason:null` (never finalized), and Claude Code appends a synthetic `user` line: `message.content:[{type:"text", text:"[Request interrupted by user]"}]` plus top-level `interruptedMessageId:<message.id>`. Confirmed by direct lookup: the interrupted assistant lines' `message.id` exactly matches `interruptedMessageId`, and the line immediately preceding the synthetic user line is an assistant line with `stop_reason:null`.

## `type:"system"`

Keys: `cwd, durationMs?, entrypoint, gitBranch, isMeta, isSidechain, messageCount?, parentUuid, sessionId, subtype, timestamp, type, userType, uuid, version` (+ subtype-specific fields).
`subtype` values found (broad scan across all session files): `api_error`, `away_summary`, `compact_boundary`, `informational`, `local_command`, `model_refusal_fallback`, `scheduled_task_fire`, `stop_hook_summary`, `turn_duration`.

- `turn_duration`: `{durationMs, messageCount}` — one per completed turn, cheap proxy for "assistant finished responding."
- `away_summary`: `{content:<string>}` — the "recap while you were away" notice (async/notification digest).
- `api_error` — **this is where transient API failures live**: `{level:"error", maxRetries, retryAttempt, retryInMs, error:{connection, formatted, message, isNetworkDown, rateLimits, requestId, status}}` — a retry-with-backoff loop, not a terminal failure by itself.
- `compact_boundary` — **context-compaction event**: `{content:<string summary>, logicalParentUuid, slug, compactMetadata:{trigger, preTokens, postTokens, durationMs, preservedMessages, preservedSegment, cumulativeDroppedTokens?, preCompactDiscoveredTools, precomputed?}}`. `logicalParentUuid` preserves logical turn-continuity across the `parentUuid` discontinuity a compaction causes — needed so a tailer's turn-tree doesn't appear to fork.
- `model_refusal_fallback` — automatic model swap after a safety refusal: `{apiRefusalCategory, apiRefusalExplanation, direction:"retry", originalModel, fallbackModel, refusedUserMessageUuid, requestId, trigger:"refusal"}`.
- `stop_hook_summary` — result of the Stop-hook run: `{hookCount, hasOutput:bool, hookErrors:[], hookInfos:[], preventedContinuation:bool, stopReason, toolUseID, hookAdditionalContext}`. `preventedContinuation:true` means a hook forced the agent to keep going instead of ending its turn.
- `scheduled_task_fire`, `local_command`, `informational`: `{content:<string>}` — cron/loop firing, a `/slash-command`-style local command result, and a generic info banner respectively.

## `type:"attachment"`

Envelope: `parentUuid, isSidechain, attachment:{type, ...}, cwd, entrypoint, gitBranch, sessionId, timestamp, uuid, userType, version`.
`attachment.type` values found (broad scan): `agent_listing_delta`, `command_permissions`, `compact_file_reference`, `date_change`, `deferred_tools_delta`, `edited_text_file`, `file`, `goal_status`, `hook_additional_context`, `hook_success`, `invoked_skills`, `mcp_instructions_delta`, `queued_command`, `skill_listing`, `task_reminder`, `task_status`.
Per-type field shapes (structure only):

- `hook_success`/`hook_additional_context`: `{hookName, hookEvent, toolUseID, content, stdout?, stderr?, command?, durationMs?, exitCode?}` — hook execution echoes (SessionStart, PreToolUse-style events).
- `deferred_tools_delta`: `{addedLines, addedNames, pendingMcpServers, readdedNames, removedNames}` — this environment's deferred-tool-loading mechanism (matches the `ToolSearch` pattern seen in this very conversation).
- `mcp_instructions_delta`: `{addedBlocks, addedNames, removedNames}`.
- `skill_listing`/`invoked_skills`: `{content, isInitial, names, skillCount}` / `{skills:[{name, path, content}]}`.
- `task_reminder`: `{content, itemCount}` — periodic re-injection of the current todo list.
- `task_status`: `{taskId, taskType, status, description, deltaSummary, outputFilePath}` — `taskType:"local_agent"` observed, i.e. this same field family tracks background-task/background-agent status deltas, not just todo items.
- `goal_status`: `{condition, met:bool, reason, sentinel?, tokens?, durationMs?, iterations?}` — output of an until-loop / wait-condition construct.
- `command_permissions`: `{allowedTools:[<toolName>]}`.
- `agent_listing_delta`: `{isInitial, addedLines:[<string>], addedTypes, removedTypes, showConcurrencyNote}` — available-agent-type roster deltas.
- `queued_command`: `{prompt, commandMode, origin, timestamp}` — mirrors `queue-operation` lines.
- `date_change`: `{newDate}` — day-rollover marker in long sessions.
- `file` / `edited_text_file` / `compact_file_reference`: IDE/editor file-tracking (`{filename, displayPath, content}` / `{filename, snippet}` / `{filename, displayPath}`).

## Other top-level line types

- `mode`: `{type, mode, sessionId}` — only `"normal"` observed (no plan-mode sample).
- `permission-mode`: `{type, permissionMode, sessionId}` — **only `"bypassPermissions"` observed across every sampled session** (1370 lines, zero other values) — see Risks.
- `queue-operation`: `{type, operation, timestamp, sessionId, content}`. `operation` values: `enqueue`, `dequeue`, `remove`, `popAll` — the input-queueing mechanism (user typing ahead while the agent works).
- `last-prompt`: `{type, leafUuid, sessionId, lastPrompt?}` — a live "most recent prompt" pointer/cache.
- `ai-title`: `{type, aiTitle, sessionId}` — the auto-generated session title shown in UI pickers.
- `bridge-session`: `{type, sessionId, bridgeSessionId:<cse_...>, lastSequenceNum}` — links a local session to a remote/mobile "bridge" session id; appears at the very start of a subset of sessions.
- `file-history-snapshot` / `file-history-delta`: `{type, messageId, snapshot:{messageId, trackedFileBackups, timestamp}}` / `{type, messageId, snapshotMessageId, trackingPath, backup:{backupFileName, version, backupTime}, timestamp}` — the "undo history" mechanism for files Claude edits; `trackingPath` names the edited file.

## Tool-use ↔ tool-result linkage (confirmed by direct lookup)

1. `assistant` line's `content[].tool_use.id` (`toolu_...`) is echoed verbatim as `user` line's `content[].tool_result.tool_use_id`.
2. Redundantly, the `user`/tool_result line's own `parentUuid` equals the `uuid` of the specific assistant line carrying that `tool_use` block (not the whole turn). Both linkages agreed in every case checked.

# 2. Subagent representation — three distinct subsystems, all confirmed by direct file inspection

`isSidechain` is **never `true`** inside a top-level `<sessionId>.jsonl`; all subagent activity lives exclusively under `<sessionId>/subagents/`. Directory-shape census across 21 top-level sessions showed 4 combinations of `{subagents/, subagents/workflows/, tool-results/, workflows/*.json}` — i.e. plain-Task, Workflow-runs, and tool-output-spill are independent, optional features per session.

## 2a. Plain synchronous/background `Agent` tool call

Spawned via `tool_use.name == "Agent"`, `input:{description, prompt, name?, model?, subagent_type?, run_in_background?}` (the human-facing "Task tool"; internal name is `Agent`, not `Task`).

- File: `<sessionId>/subagents/agent-<agentId>.jsonl`; sidecar `<sessionId>/subagents/agent-<agentId>.meta.json`.
- `agent-*.jsonl` schema = **same envelope as the main transcript** (`type:user|assistant`, `message`, `parentUuid`, `uuid`, `timestamp`, ...) plus extra fields: `agentId`, `promptId` (user lines), `attributionAgent` (assistant lines, value `"workflow-subagent"` seen), `slug` (a human-readable random tag, e.g. three-word slug). `isSidechain:true` on every line. `sessionId` inside the agent file equals the **parent** session's id, not a new id.
- Two `meta.json` shapes observed:
  - **Unnamed/synchronous** (`agentType` often `"general-purpose"`): `{agentType, description, toolUseId:<toolu_id>, spawnDepth}` — **`toolUseId` is a direct, structured cross-reference to the exact `Agent` tool_use in the parent (or an ancestor agent's) transcript.**
  - **Named/backgroundable "teammate"** (`taskKind:"in_process_teammate"`): `{agentType, description, name, spawnDepth, taskKind, teamName, color, model, permissionMode, planModeRequired}` — **no `toolUseId`.** Cross-reference is instead: the spawning `Agent` tool_use has `input.name == meta.json.name`, and its `tool_result` text contains a free-text line `agent_id: <name>@<teamName>` (unstructured — needs parsing).
- `spawnDepth` observed values 0, 1, 2 — subagents can spawn subagents; nested agents are stored **flat** in the same `subagents/` dir (no per-parent subfolder) and have no `parentAgentId` field — the only way to find a nested agent's spawner is to match its `meta.json.toolUseId` against `Agent` tool_use ids found in ANY sibling `agent-*.jsonl`, not just the top-level transcript.
- Peer/background agent output reaching the spawning session appears as an ordinary `user` turn there with `origin:{kind:"peer", from:<agentName>, senderTaskId:<agentId>}`.

## 2b. Named-teammate "team" registry (persistent, cross-session, mailbox-based)

`teamName` (e.g. `"session-4f5f2654"`) does **not** necessarily name the current session — confirmed directly: an agent spawned inside session `03d4c2bf...` carried `teamName:"session-4f5f2654"`, and `~/.claude/teams/session-4f5f2654/config.json` has `leadSessionId:"4f5f2654-affb-4ba7-bd29-427968027bab"` — a **different, earlier** top-level session that originally created the team. Teams persist and can be resumed/reused across unrelated later sessions (consistent with recurring/cron-driven multi-agent workflows).

- `~/.claude/teams/session-<8hex>/config.json`: `{name, createdAt, leadAgentId, leadSessionId, members:[{agentId:"<name>@<teamName>", name, agentType, color?, model?, joinedAt, tmuxPaneId, cwd, subscriptions:[], backendType:"in-process", planModeRequired?, prompt?}]}`. `prompt` holds the full system prompt given to that teammate — **highly sensitive, structure-only in any adapter.**
- `~/.claude/teams/session-<8hex>/inboxes/<agentName>.json`: array of mailbox messages, `{from, to?, type:"message", text, timestamp, color, msgV, msg_id, read:bool}` — this is the persisted state behind the `SendMessage` tool (`tool_use.input:{content, message, recipient, summary, to, type}`). `read:false` entries are effectively pending/unconsumed notifications.

## 2c. Multi-agent `Workflow` tool runs (fan-out/fan-in orchestration DSL)

Spawned via `tool_use.name == "Workflow"`, `input:{script|scriptPath, args?, description?, run_in_background?}` (`script` is an embedded JS/TS-like DSL defining `phases`, per-agent prompts, and a JSON-schema for structured results).
Cross-reference: the `Workflow` tool_use's `tool_result` is **unstructured text** containing `Task ID: <taskId>`, a one-line `Summary`, `Transcript dir: <absolute path to subagents/workflows/wf_<id>>`, and `Script file: <path>` — a tailer must regex-parse this (or, more robustly, fs-watch `subagents/workflows/` for new directories).
Layout per run, `<sessionId>/subagents/workflows/wf_<runId>/`:

- `agent-<agentId>.jsonl` — **identical schema to 2a's agent files**, `agentId` matches hex id used in `workflowProgress[].agentId` (top-level run summary) and in `journal.jsonl`.
- `agent-<agentId>.meta.json` — minimal here: always exactly `{"agentType":"workflow-subagent"}` (no `toolUseId`, no `name`; correlation is purely by filename/`agentId`).
- `journal.jsonl` — append-only memoization log: `{type:"started", key:"v2:<hash>", agentId}` then later `{type:"result", key, agentId, result:<schema-shaped object>}`. `key` is a content hash of the agent's task, used to skip re-running identical work on resume/retry (5 `started` + only 4 or 5 `result` in samples — a missing `result` line is exactly the observable signature of an agent that failed to call `StructuredOutput`).
- Top-level `<sessionId>/workflows/wf_<runId>.json` — **a terminal summary written once, at the end of the run** (confirmed by mtime ordering: agent files finish first, `journal.jsonl` next, this file last, ~10 min after). Keys: `runId, timestamp, taskId, script, scriptPath, result, agentCount, logs:[<string>], durationMs, summary, workflowName, status, startTime, phases:[{title}], defaultModel, workflowProgress:[...], totalTokens, totalToolCalls`. `status` values seen across 29 run-summary files: `completed`(23), `failed`(4), `killed`(2). `workflowProgress[]` entries: `{type:"workflow_phase", index, title}` and `{type:"workflow_agent", index, label, phaseIndex, phaseTitle, agentId, model, state, startedAt, queuedAt, attempt, lastToolName, lastToolSummary?, promptPreview, lastProgressAt, tokens, toolCalls, durationMs, resultPreview?}` — `state` seen: `done`, `error` (others such as `running`/`queued` plausible but not observed). **Implication: during an in-flight workflow the run-summary JSON does not exist/update yet — live progress must come from the `agent-*.jsonl` files (which flush incrementally, same guarantee as main transcripts) and `journal.jsonl`.**

## 2d. `tool-results/` spill directory (per top-level session, `<sessionId>/tool-results/`)

Any tool result too large to inline gets written here and the transcript keeps only a pointer/summary. Two filename conventions, both confirmed:

- Generic (Bash, etc.): `b<8-char base36 id>.txt` — random id, no tool name embedded; referenced in `toolUseResult.persistedOutputPath` and in a boilerplate `"<persisted-output>... Full output saved to: <path>"` message.
- MCP tools: `mcp-<serverName>-<toolName>-<epochMs>.txt` — referenced in a boilerplate `"Error: result (<N> characters) exceeds maximum allowed tokens. Output has been saved to <path>..."` message, which also states the JSON schema of the spilled file inline.

# 3. Liveness (directly verified, not inferred)

Two `stat` calls on the live session file ~2 minutes apart (intervening work happened in between): size grew 351,133 → 412,763 bytes, line count 139 → 158. Cross-checked against `~/.claude/sessions/39911.json` (below): `sessionId` for that exact PID matched the live session and `status:"busy"`, `updatedAt`/`statusUpdatedAt` timestamps moving forward — **direct, non-inferential confirmation that transcripts are appended incrementally, line-by-line, while a session is actively running**, and that a much cheaper liveness signal than mtime-polling exists (see next section).

# 4. Cross-session index / registry files (outside any project directory)

- **`~/.claude/sessions/<pid>.json`** — live process registry, one file per running Claude Code process, named by OS PID. Keys: `pid, sessionId, cwd, startedAt, procStart, version, peerProtocol, kind ("interactive"), entrypoint, name, nameSource, status, updatedAt, statusUpdatedAt` (+ optional `bridgeSessionId`). `status` values observed: `"idle"`, `"busy"`. **This is the correct liveness/discovery primitive for a tailer** — directly confirmed: the task's designated "current active session" (`<SESSION_ID>`) appeared here with `status:"busy"` while this investigation ran, and other concurrently-open sessions showed both `idle` and `busy`. One file (of 8 sampled) lacked `status`/`updatedAt`/`statusUpdatedAt` entirely — a just-started process before its first status push; a tailer should treat missing `status` as "unknown/starting."
- **`~/.claude/history.jsonl`** (global, cross-project, single file, 8,894 lines in this install) — one line per submitted prompt across every project: `{display:<string>, pastedContents:<object>, project:<path>, sessionId:<uuid>, timestamp:<epoch-ms>}`. Confirms cross-project prompt log exists but is **not** scoped to `<FLATCWD>` alone.
- **`<projectDir>/sessions-index.json`** (per-project, e.g. `.../projects/<FLATCWD>/sessions-index.json`) — `{version, entries:[{sessionId, fullPath, fileMtime, firstPrompt, messageCount, created, modified, gitBranch, projectPath, isSidechain}], originalPath}`. In this install it was **stale** (last modified weeks before the newest sessions, only 3 old entries) — **not a reliable live index; do not depend on it being current.**
- **`~/.claude/tasks/session-<8hex>/`** — only `.lock` (empty, flock target) and `.highwatermark` (a single integer, todo-list id counter) were found; the actual todo-item content is **not** snapshotted here and must be reconstructed by replaying `TaskCreate`/`TaskUpdate` tool calls from the transcript.
- **`~/.claude/teams/session-<8hex>/`** — see 2b above.

# 5. Privacy note on what was touched

Several files inspected during this investigation (workflow `script` bodies, teammate `prompt` fields, `history.jsonl` `display` text, `queue-operation.content`, `last-prompt.lastPrompt`, tool_result bodies) contain real user-authored text, file contents, resume/job-search PII, and literal shell/SQL/command strings. None of that is reproduced above — every field is reported as name+type/shape only, per the privacy rule.

## RISKS

- **`approval_requested` is entirely unverified.** Every one of the ~20 sessions sampled ran under `permission-mode:"bypassPermissions"` (1,370/1,370 `permission-mode` lines). No interactive approval-prompt shape was ever produced, so its on-disk representation (if any exists at all beyond simply delaying the `tool_result`) is a documented gap, not a documented fact.
- **`mode` was only ever seen as `"normal"`.** Plan mode, accept-edits mode, etc. exist as product concepts (tool names like `ExitPlanMode` appear in passing) but no transcript in this corpus exercised them, so their transcript shape is unconfirmed.
- **`workflowProgress[].state` vocabulary is incomplete.** Only `done` and `error` were observed; `running`/`queued`/similar in-flight states are plausible (workflows are provably in-flight at some point, per the mtime-ordering evidence) but never captured on disk in this sample because the run-summary file is written only at the end.
- **`sessions-index.json` is stale in this install** (weeks old, 3 entries) and should not be relied upon as a live index — this may be specific to this machine's cleanup/GC cadence rather than a general guarantee; a target install should be re-checked.
- **The `teamName`-to-session resolution logic is inferred from one example.** It's a strong, directly-verified data point (`leadSessionId` in `config.json` matched a genuinely different, earlier session than the one referencing it), but only one team was traced end-to-end; edge cases (team deleted, team owned by a session outside the current project dir, concurrent teams with colliding short-prefixes) were not tested.
- **`caller` on `tool_use` blocks was only ever seen as `{"type":"direct"}`** — the field clearly anticipates other values (indirect calls, e.g. from a hook or a queued command) that never appeared in this corpus.
- **File sizes**: several source transcripts are 30–115MB; all sampling here used `jq`/`grep`/`head`/`tail` streaming and targeted `sed -n`, never a full read, per the task's instruction — but a production tailer must assume files can be much larger still and should never buffer a whole session file in memory.
- **No `redacted_thinking` content block was observed** in this data, though it's a documented possibility in the underlying API; a production parser should still handle it defensively.

## RECOMMENDATIONS

- **Tail at the JSONL-line level, buffer by `message.id`.** Do not emit an "assistant message" event per JSONL line — buffer consecutive `assistant` lines sharing `message.id`, concatenate `content[]` in arrival order, and only emit once a line in that group has non-null `stop_reason` (or the group is orphaned by a later `interruptedMessageId`, in which case emit a truncated/interrupted event instead).
- **Use `~/.claude/sessions/<pid>.json` as the primary liveness/discovery source**, not `.jsonl` mtime polling. It's O(1) to check `status`, gives you the `sessionId`↔`pid`↔`cwd` mapping for free, and directly distinguishes "actively generating" (`busy`) from "sitting idle" (`idle`) — mtime polling can't tell those apart during a long tool call.
- **Treat subagent discovery as filesystem-watch, not transcript-parse, wherever possible.** fs-watching `subagents/` and `subagents/workflows/` for new `.jsonl`/`.meta.json` files is more robust than regexing tool_result text (the `Workflow` tool_result text format is not a stable structured contract) and than trusting `toolUseId` alone (absent for named teammates).
- **Build two correlation paths for subagents, not one:** (a) structured `toolUseId` match for plain/synchronous `Agent` calls — reliable; (b) `name`/`teamName`-based glob + `origin.kind` sniffing for named/background teammates — inherently fuzzier, and `teamName` may point at an _ancestor_ session's directory, so the adapter must be able to resolve `teams/session-<8hex>/` even when `<8hex>` isn't the current session's own prefix.
- **Recurse into every discovered `agent-*.jsonl` when resolving nested (`spawnDepth>0`) subagents.** There is no `parentAgentId`; the only way to attribute a `spawnDepth:2` agent to its spawner is to search all sibling agent transcripts (not just the top-level session file) for a `tool_use.id` matching its `meta.json.toolUseId`.
- **Don't wait on `workflows/wf_*.json` for live workflow progress** — it's written once, at the very end. Drive the UI from `subagents/workflows/wf_*/agent-*.jsonl` (incremental) and `journal.jsonl` (per-agent completion) instead, and only use the top-level summary JSON for final rollup/history views.
- **Spool-aware tool-result rendering**: detect `toolUseResult.persistedOutputPath` (Bash) and the `mcp-<server>-<tool>-<epochMs>.txt` naming convention to offer "show full output" without re-running the tool.
- **Explicitly design for the `approval_requested` gap.** Since this repro environment can't produce a real approval prompt, either (a) ship the adapter's approval/pending-permission indicator as best-effort/experimental and validate against a real interactive (non-bypass) session before launch, or (b) derive it purely from configuration (`permission-mode` line + tool allow/deny lists in `command_permissions` attachments) rather than from a transcript event that may not exist.
- **Privacy/security posture for the product itself**: several files a tailer will legitimately need to read (`teams/*/config.json` `.members[].prompt`, `history.jsonl` `.display`, tool_result bodies, workflow `.script`) routinely contain the user's real, sensitive content (PII, resumes, business data, literal shell/SQL commands). The adapter's storage/transport layer should treat everything it reads as sensitive by default and never assume "it's just metadata."
